认识我们的角色

EM-BOX 系统由 5 个核心角色协作运行。每个角色都有自己的职责、颜色和性格。理解它们的关系，你就理解了整个系统。

🎯
Frontend (React)
src/
系统的"脸面"。负责展示界面、捕获用户点击、管理表单状态。使用 React 19 + TypeScript + Tailwind CSS。所有 UI 文案都是中文。
📡
API Client
src/shared/lib/apiClient.ts
前后端之间的"信使"。自动附加 JWT 令牌、处理 30 秒超时、在令牌过期时自动刷新。它是前端唯一的 HTTP 出口，所有模块的请求都经过它。
⚡
Express Backend
server/src/
系统的"大脑"。接收请求、验证权限、构建 SQL 查询、调用 AI 服务。使用 Express 4 + PostgreSQL。每个模块都有对应的路由文件。
🗃
PostgreSQL Database
database.ts (pg Pool)
数据的"金库"。存储候选人、职位、面试记录等所有结构化数据。使用 JSONB 字段存储评分配置、等级规则等灵活数据。
🤖
AI Services
server/src/modules/ai/
系统的"预言家"。支持 OpenAI、Anthropic、智谱、DeepSeek 等 8 个 LLM 提供商。负责简历评分、面试打分、候选人排名。
关键设计决策 前端使用 camelCase，后端使用 snake_case。这不是随意选择 — 它遵循各自生态的最佳实践。Mapper 函数是两者之间的"翻译官"。
场景：产品经理说"AI 评分不准"。根据角色分工，这个问题最可能出在哪里？

Mock vs Real API Switch

每个 API 函数都有两条路径：如果 USE_MOCK_API 为 true（开发模式下默认开启），直接返回内存中的假数据，不需要后端运行。

切换到真实 API 只需在 .env 中设置 VITE_USE_MOCK_API=false，无需修改任何业务代码。这种设计让前端可以独立开发和测试。

Code // auth.ts — 生成令牌对 export function generateTokenPair( payload: { userId: string; email: string; role: string; } ) { const jti = crypto.randomUUID(); const accessToken = jwt.sign( payload, env.JWT_SECRET, {expiresIn: env.JWT_EXPIRES_IN} ); const refreshToken = jwt.sign( {...payload, jti, type: 'refresh'}, env.JWT_SECRET, {expiresIn: env.JWT_REFRESH_EXPIRES_IN} ); return {accessToken, refreshToken, jti}; }
English

JWT Token Pair

系统使用双令牌机制：Access Token（短期，24小时）和 Refresh Token（长期，7天）。

每个令牌都有唯一的 jti（JWT ID），用于 令牌黑名单追踪。用户登出时，令牌被加入黑名单，即使未过期也无法使用。

为什么用双令牌？ Access Token 频繁使用，有效期短更安全。Refresh Token 只在 Access Token 过期时使用一次，有效期可以长一些。这样既保证了安全性，又不会让用户频繁重新登录。
场景：前端开发者在本地开发，不想启动后端。他应该怎么做？

English

Prompt Engineering

System Prompt 是 AI 评分的核心。它定义了 AI 的角色（"专业招聘评估师"）、评分维度（从数据库的 scoring_rules 读取）和输出格式（严格的 JSON）。

Module 05

当出错了怎么办

没有系统能永远不出错。关键在于：出错了怎么处理？EM-BOX 有一套完善的错误处理机制，从令牌过期到数据库约束冲突，每个场景都有预案。

Code // errors.ts — 错误类层级 export class AppError extends Error { constructor( public statusCode: number, message: string, public code?: string, ) { super(message); } } export class NotFoundError extends AppError { constructor(resource: string) { super(404, `${resource} not found`, 'NOT_FOUND'); } } export class ValidationError extends AppError { constructor(message: string) { super(400, message, 'VALIDATION_ERROR'); } }
English

Error Class Hierarchy

所有业务错误都继承自 AppError，每个子类预设了对应的 HTTP 状态码和错误码。

NotFoundError → 404、UnauthorizedError → 401、ForbiddenError → 403、ValidationError → 400。这样后端只需 throw new NotFoundError('Candidate')，错误处理中间件会自动返回正确的 HTTP 响应。

Code // errorHandler.ts export function errorHandler( err: Error, _req: Request, res: Response, _next: NextFunction ) { if (err instanceof AppError) { res.status(err.statusCode).json({ error: { code: err.code, message: err.message, }, }); return; } // PostgreSQL 错误码映射 if (pgCode === '23505') { // 409 DUPLICATE } if (pgCode === '23503') { // 400 FK_VIOLATION } // 兜底: 500 INTERNAL_ERROR }
English

Centralized Error Handler

Express 的错误处理中间件接收所有未捕获的错误。它首先检查是否是 AppError 子类 — 如果是，直接用预设的状态码响应。

然后检查 PostgreSQL 错误码：23505 表示唯一约束冲突（重复数据），23503 表示外键约束违反（引用的记录不存在）。这些都被映射为有意义的 HTTP 状态码。

Code // apiClient.ts — 令牌自动刷新 if (response.status === 401 && token) { const errorData = await response .json().catch(() => ({})); if (errorData?.error?.code === 'TOKEN_EXPIRED' || errorData?.error?.code === 'TOKEN_REVOKED') { const newToken = await tryRefreshToken(); if (newToken) { // 用新令牌重试请求 headers.set('Authorization', `Bearer ${newToken}`); response = await fetch(...); } else { // 刷新失败，强制重新登录 localStorage.removeItem( 'em-box.auth-token'); window.location.reload(); } } }
English

Silent Token Refresh

当 API 返回 401 且错误码是 TOKEN_EXPIRED 或 TOKEN_REVOKED 时，fetchJson 不会直接报错，而是自动尝试用 Refresh Token 获取新的 Access Token。

如果刷新成功，用新令牌重试原始请求 — 用户完全无感知。如果刷新也失败（Refresh Token 也过期了），则清除所有认证信息并刷新页面，强制用户重新登录。

Code // csrf.ts — CSRF 防护 export function csrfMiddleware( req: Request, res: Response, next: NextFunction ) { // GET/HEAD/OPTIONS 安全方法直接放行 if (['GET','HEAD','OPTIONS'] .includes(req.method)) { next(); return; } // Bearer Token 认证已天然防 CSRF const auth = req.headers.authorization; if (auth?.startsWith('Bearer ')) { next(); return; } // 其他情况检查 Origin 头 // ... 白名单校验 }
English

CSRF Protection

CSRF 是一种常见的 Web 攻击。但由于 EM-BOX 使用 Bearer Token（存在 localStorage 中，而非 Cookie），天然免疫传统 CSRF 攻击。

中间件仍添加了 Origin 头检查作为深度防御 — 对于非 Bearer Token 的修改请求，会验证来源域名是否在白名单中。

错误处理的黄金法则 后端抛出有意义的错误码（NOT_FOUND, TOKEN_EXPIRED, DUPLICATE...），前端根据错误码做对应处理（重试、提示、跳转登录）。不要只看 HTTP 状态码 — 业务错误码才是关键。
场景：用户操作时突然被踢回登录页。后端日志显示"TOKEN_REVOKED"。最可能发生了什么？
